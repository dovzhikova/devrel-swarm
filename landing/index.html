Your editorial standards belong in git, not a prompt

The standard answer to “the AI is off-voice” is to keep editing the system prompt until you forget what was in there last week. devrel-swarm puts the editorial contract in three small files — voice.md, style.md, slop-blocklist.md — that live in .devrel/ next to your source. Diff them. Branch them. Review them like a PR. 

$ devrel init --name myproduct --url https://myproduct.dev --github-repo me/myproduct
+ .devrel/
+ kb/
+ deliverables/
+ context/
+ config.toml          # product identity, model selection, budget caps
+ voice.md             # tone profile + sample passages    (commit)
+ style.md             # house style + per-content targets (commit)
+ slop-blocklist.md    # banned phrases                    (commit)
+ .gitignore
+ state.db
Done. Edit voice.md / style.md / slop-blocklist.md, then run devrel doctor.

When the output drifts off-voice, you add a sample to voice.md and commit. The next run picks it up. Every editorial decision your project ever makes is a git hash with a diff. Yes, putting your editorial standards in version control is as obvious as it sounds. We can't explain why nothing else does it.

Eight critique stages, not one prompt

ChatGPT-style content tools generate a draft, ask the model “is this good?”, accept the answer, and ship. devrel-swarm routes every piece through eight discrete stages with explicit pass/fail criteria, force-rewrite loops on failure, and a complete revision trace.

draft → developmental edit → line edit → copy edit → anti-slop (regex + LLM lint, force-rewrite once or AbortLoud) → persona (skeptical-dev scorer) → readability (Flesch-Kincaid + sentence stats) → re-loop into copy edit if persona / readability fails → final draft + revision-trace.json

Every stage's score and diff goes into a JSON trace. You can read what the developmental editor flagged (“hook is buried in paragraph 4”), what the persona scorer pushed back on (“you'd lose me at sentence two — too marketing”), which sentences the readability pass forced shorter. Slop doesn't survive contact with seven critics. And you can prove it with a diff.

Stages 1-4 use Sonnet; the cheaper stages use Haiku. Total cost lands at roughly 2.5-4× a single revision loop.

Argus tells you what's working — and what to kill

Generating content is the part every tool does. Knowing what to do with what already shipped is the part that actually moves a metric — and the part LLMs are bad at on their own, because they have no project history. Argus pulls per-piece metrics from PostHog (page views, conversions), GitHub (stars delta, issue velocity), Instantly (reply rates), and Echo's social_mentions table; ranks them deterministically; and only then asks Sonnet for a typed Recommendation. The output is a dataclass, not a paragraph of advice.

  • You stop guessing what to write next. Six typed actions: double_down, retire, rewrite, retest, amplify, investigate. No “optimize”, no “explore opportunities”, no other LLM hand-waving. Below confidence 0.5 it's always investigate — the prompt enforces it.
  • You stop guessing whether Argus is right. After a few cycles, devrel analytics calibration reports per-action hit rate, lift vs coin-flip, and separates high-confidence (≥0.8) from low-confidence (<0.5) accuracy. If the model isn't predictive for your project, you see it in numbers, not vibes, and you can ignore it accordingly.
  • You stop keeping a metrics spreadsheet. metric_history(content_id, period_end, primary_metric), PRIMARY KEY on (content_id, period_end), index on (content_id, period_end DESC). Week-over-week is one SELECT. devrel analytics history blog/x drills into any piece.

FAQ

Does it lock me in?

No backend. The CLI runs locally. Your data is in .devrel/state.db and knowledge_base/. Walk away anytime — there's nothing to unsubscribe from.

Why is config in git?

Same reason your CI config is in git: editorial standards are decisions, and decisions belong in source. The four committed files (config.toml, voice.md, style.md, slop-blocklist.md) diff and review like any other source.

What if I already use HubSpot or Buffer?

devrel-swarm produces the content. Paste it where you already work. The point isn't to replace your distribution stack — it's to replace the part where you sit down to write the post in the first place.

Does it work without all the integrations?

Yes. PostHog, GitHub, Instantly, Apollo, Telegram, Sheets — each is optional. Missing keys cause graceful degradation, not crashes. Argus marks the failed source unhealthy and ships a partial report.

Is it production-ready?

800 passing tests, multiple shipped versions, a 13-agent system maintained by one person. Kick the tires on a side project before pointing it at your main repo. It's also free, so the worst case is you wasted an evening.

Is there an enterprise tier?

No.

How is this different from [shiny new AI marketing tool]?

The other ones are SaaS, write one draft per piece, and don't measure what shipped. This is a CLI, runs eight critique stages per piece, and a calibration loop on its own recommendations. If those differences don't matter to you, the other ones are fine and you should stop reading this.
